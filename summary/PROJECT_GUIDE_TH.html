1. 20x คืออะไร และมีไว้ทำไม

ปัญหา: เครื่องมือ AI ส่วนใหญ่บังคับให้เรา copy-paste context ไปมาระหว่างแท็บ เช่น copy ticket จาก Jira ไปวางใน ChatGPT แล้ว copy โค้ดกลับมาวางใน editor ซึ่งเสียเวลามหาศาล

แนวคิดของ 20x: กลับด้านความคิด "ให้งานวิ่งไปหา AI แทนที่เราจะป้อนงานให้ AI"

3

3. สถาปัตยกรรม Electron (เทียบกับ Django)

Electron app ประกอบด้วย 2 process ที่แยกกันเด็ดขาด ตรงนี้สำคัญที่สุดสำหรับคนมาจาก backend:

┌──────────────────────────────────────────────────────────────┐
│  Renderer Process (React UI)         ← เหมือน "Frontend/SPA"  │
│  ❌ ไม่มีสิทธิ์แตะ Node.js / ไฟล์ / database โดยตรง            │
├──────────────────────────────────────────────────────────────┤
│  Preload Bridge (src/preload)        ← เหมือน "API Gateway"   │
│  ✅ whitelist ว่า UI เรียกอะไรได้บ้าง (window.electronAPI)     │
├──────────────────────────────────────────────────────────────┤
│  Main Process (Node.js)              ← เหมือน "Django Backend"│
│  ✅ SQLite, file system, spawn agent processes, HTTP servers  │
└──────────────────────────────────────────────────────────────┘

Data Flow:  React UI → Zustand → IPC Client → Preload → Main → SQLite
🐍 เทียบ Django ตรงๆ
Django20x (Electron)
views.py + DRF ViewSetssrc/main/ipc-handlers.ts
models.py + ORMsrc/main/database.ts (SQL ตรงๆ)
migrations/runMigrations() + SCHEMA_VERSION
Celery workersagent-manager, heartbeat-scheduler, recurrence-scheduler
urls.pyชื่อ IPC channel เช่น db:getTasks, agentSession:start
DRF SerializersTypeScript interfaces
settings.pyตาราง settings (key-value)
Django Apps (3rd party)Plugins ใน src/main/plugins/

ความปลอดภัย: เปิด contextIsolation: true + nodeIntegration: false แปลว่าโค้ด React ไม่มีทางแตะ DB ได้เอง ต้องผ่าน bridge เท่านั้น (เหมือน frontend ที่เรียกได้แค่ REST API)

5. Database Schema (SQLite)

ทุกอย่างอยู่ใน SQLite ไฟล์เดียวในเครื่อง (โหมด WAL) Schema ทั้งหมดอยู่ใน src/main/database.ts

🐍 มุมมอง Django Dev

ไม่มี ORM ไม่มีไฟล์ migration แยก ใช้ createTables() สำหรับติดตั้งใหม่ + runMigrations() เช็ค SCHEMA_VERSION แล้ว ALTER TABLE ทีละขั้น (มีคู่มือใน docs/database-migrations.md)

ตารางหน้าที่

agentsคอนฟิก agent: coding_agent (claude-code/opencode/codex), model, system_prompt, mcp_servers, skill_ids

6. วงจรชีวิตของ Task (Task Lifecycle)

Task มี 6 สถานะ (ประกาศใน src/shared/constants.ts):

not_started → triaging → agent_working → ready_for_review → completed ⇄ agent_learning

จากไปเงื่อนไข
(สร้างใหม่)not_startedสร้างจาก UI, MCP, sync หรือ API
not_startedtriagingauto-run เปิด + งานยังไม่มี agent_id
triagingnot_startedtriage เสร็จ (agent_id ถูกกำหนดแล้ว)
not_startedagent_workingauto-run หยิบไปทำ หรือสั่งรันเอง
agent_workingready_for_reviewagent ทำเสร็จ (session idle)
ready_for_reviewcompletedผู้ใช้กด "Complete Task"
completedagent_learningผู้ใช้ส่ง feedback (ดาว + คอมเมนต์)
agent_learningcompletedสกัด skill เสร็จ

Auto-Run (ปุ่ม Play ที่ sidebar)

  • Scheduler (use-agent-auto-start.ts) หา task ที่พร้อม: not_started + มี agent_id + ไม่ถูก snooze
  • จัดกลุ่มตาม agent เรียงตาม priority (critical > high > medium > low)
  • แต่ละ agent รันพร้อมกันได้ตาม max_parallel_sessions (default 1) ที่เหลือเข้าคิว
  • เช็คซ้ำทุก 60 วินาทีกันงานค้าง
🐍 เทียบ Django

นี่คือ Celery worker + task queue ฉบับย่อส่วนที่รันใน process เดียว

7. ระบบ AI Agent Orchestration (หัวใจของโปรเจค)

ไฟล์หลัก: src/main/agent-manager.ts (ใหญ่ที่สุดในโปรเจค ~188KB)

Adapter Pattern : รองรับ AI 3 ค่าย

CodingAgentAdapter (interface กลาง: coding-agent-adapter.ts)
├── claude-code-adapter.ts  → Claude Code (@anthropic-ai/claude-agent-sdk)
├── opencode-adapter.ts     → OpenCode (@opencode-ai/sdk, spawn server แยก)
└── acp-adapter.ts          → Codex ของ OpenAI (@zed-industries/codex-acp)

methods: createSession() / resumeSession() / streamSession()
         abortSession() / approve() / stop()
🐍 เทียบ Django

เหมือน abstract class PaymentGateway ที่มี StripeGateway, OmiseGateway สลับกันได้โดย business logic ไม่ต้องรู้ว่าคุยกับเจ้าไหน

Session Lifecycle 6 ขั้น

1
STARTเตรียม git worktree → เขียนไฟล์ skills ลง workspace → สร้าง session ผ่าน adapter → ส่ง prompt
2
STREAMINGPolling Coordinator กลาง poll ทุก 2 วิ → dedup → ส่งให้ UI ผ่าน IPC agent:output (มี watchdog abort session ที่ค้างเกิน 5 นาที)
3
HITLagent ขอ approve สิ่งเสี่ยง → UI แสดง banner → Approve/Reject (+ ข้อความ) → ส่งกลับให้ agent
4
COMPLETEagent idle ครบ grace period → status เป็น ready_for_review → แจ้ง UI
5
LEARNINGผู้ใช้ให้ดาว + คอมเมนต์ → ส่งเข้า session เดิม → agent ปรับ SKILL.md → sync เข้า DB
6
ABORT/STOPผู้ใช้สั่งหยุดเมื่อไหร่ก็ได้ → cleanup session
💡 Use Case: ทำไมต้องมี "Polling Coordinator กลาง"?

ปัญหา: ถ้ามี 10 sessions แล้วแต่ละตัวตั้ง setInterval เอง จะมี timer 10 ตัว ยิง DB writes พร้อมกันจน event loop ของ Node.js สำลัก (UI ค้าง)

ทางแก้: timer ตัวเดียว วนทุก 2 วินาที ไล่ poll ทุก session ตามลำดับ พร้อม cap ขนาด dedup structures ที่ 5,000 รายการกัน memory บวม (เทียบ Django: เหมือนรวม cron ยิบย่อยเป็น Celery Beat ตัวเดียวที่ dispatch อย่างมีระเบียบ)

8. Triage Agent : ระบบคัดแยกงานอัตโนมัติ

เมื่องานใหม่เข้ามาโดยยังไม่รู้ว่า agent ตัวไหนควรทำ ระบบจะส่ง AI คัดแยกไปวิเคราะห์ก่อน:

งานใหม่ (ไม่มี agent_id) → status: 'triaging' → default agent ตื่นขึ้นมา
   → find_similar_tasks  (ค้นงานเก่าที่คล้ายกันด้วย SQL LIKE)
   → list_agents / list_skills / list_repos
   → update_task(agent_id, skill_ids, repos, priority, labels) ครั้งเดียว
→ status กลับเป็น 'not_started' (มี agent_id แล้ว) → auto-run หยิบไปรันจริง
💡 Use Case จริง

ทีมคุณมี agent 2 ตัว: "Backend Agent" (ถนัด Django/API) กับ "Frontend Agent" (ถนัด React) มี ticket ใหม่จาก Linear: "แก้ bug: API /orders ตอบ 500 เมื่อส่ง date format ผิด"

Triage agent ค้นงานเก่าด้วยคีย์เวิร์ด "API", "500", "orders" พบว่างานแบบนี้ Backend Agent เคยทำสำเร็จ 5 ครั้ง จึงมอบหมาย Backend Agent + แนบ skill "django-error-handling" + repo "company/backend-api" + ตั้ง priority เป็น high แล้วถอยออกไปให้ auto-run ทำงานต่อ

Safety Guards

  • API /update_task บล็อกการเปลี่ยน status ระหว่างอยู่สถานะ triaging (กัน agent มือลั่น)
  • Retry สูงสุด 2 ครั้ง ถ้ายังหา agent ไม่ได้จะ toast ให้มนุษย์เลือกเอง
  • มีปุ่ม "Triage" (ไอคอนประกาย ✨) ให้กดสั่งเองในหน้า task detail

9. ระบบ Skills และ Auto-Learning

Skill = คลังความรู้/คำสั่งสำเร็จรูปที่ agent โหลดไปใช้ตอนทำงาน เก็บเป็น Markdown:

---
name: django-migration-safety
description: แนวทางเขียน migration อย่างปลอดภัยใน production
---

## ขั้นตอน
1. ห้าม drop column ทันที ให้ deprecate ก่อน
2. ...
  • ตอนเริ่ม session ระบบเขียน skills เป็นไฟล์: workspaces/<taskId>/.agents/skills/<name>/SKILL.md
  • ลำดับการเลือก: ระดับ task > ระดับ agent > ไม่กำหนด (โหลดทั้งหมด)
  • แต่ละ skill มี version (เพิ่มอัตโนมัติ), confidence (0-1), ตัวนับ uses

Auto-Learning Loop (ฟีเจอร์เด็ดสุด)

งานเสร็จ → ให้ดาว 1-5 + คอมเมนต์
   → learnFromSession() ส่ง feedback เข้า session เดิม (status: agent_learning)
   → agent ทบทวนงานตัวเอง แก้/สร้าง SKILL.md ใน workspace
   → syncSkillsFromWorkspace() สแกนกลับเข้า DB (version +1, ปรับ confidence)
💡 Use Case จริง

Agent ทำงาน "เพิ่ม endpoint ใหม่" แล้วลืมเขียน test คุณให้ 3 ดาวพร้อมคอมเมนต์ "โค้ดโอเค แต่ลืม unit test ทุกครั้งเลย"

ระบบส่ง feedback เข้า session เดิม agent อัปเดต skill "api-development" เพิ่มหัวข้อ "ต้องเขียน unit test ครอบ endpoint ใหม่เสมอ" → จากนี้ agent ทุกตัวที่โหลด skill นี้ จะไม่ลืม test อีก ระบบฉลาดขึ้นเองจาก feedback ของเรา

10. Git Worktree Management

ปัญหา: agent 3 ตัวทำ 3 tasks ใน repo เดียวกันพร้อมกัน ถ้าใช้ clone เดียวกัน = แก้ไฟล์ชนกันพินาศ

ทางแก้ (src/main/worktree-manager.ts):

~/<userData>/
├── repos/                    ← bare clone (โหลดครั้งเดียว แชร์ git objects)
│   └── <org>/<repo>.git
└── workspaces/               ← worktree แยกต่อ task
    ├── <taskId-1>/<repo>/   ← branch: task/<taskId-1>
    └── <taskId-2>/<repo>/   ← branch: task/<taskId-2>
  • clone ด้วย gh repo clone --bare (GitHub) หรือ glab repo clone --bare (GitLab)
  • แต่ละ task ได้ branch + โฟลเดอร์ของตัวเอง ประหยัด disk เพราะแชร์ git objects
  • จบงานแล้ว git worktree remove --force เก็บกวาดสะอาด
🐍 เทียบ Django

เหมือนแต่ละ test case ได้ database transaction แยกของตัวเอง ทำอะไรก็ไม่กระทบคนอื่น

12. IPC Architecture : Frontend คุยกับ Backend ยังไง

ส่วนที่ backend dev ควรเข้าใจที่สุด เพราะมันคือ "REST API ภายในแอป"

แบบที่ 1: Request-Response (เหมือน REST)

React component
→ window.electronAPI.db.getTasks()
→ ipcRenderer.invoke('db:getTasks')
→ ipcMain.handle('db:getTasks')
   ทำงานกับ SQLite
→ return Promise กลับ React

แบบที่ 2: Event Streaming (เหมือน WebSocket)

// main ยิงฝั่งเดียว:
webContents.send('agent:output', data)

// renderer subscribe:
ipcRenderer.on('agent:output', cb)

ใช้กับ output ของ agent แบบ real-time

ตั้งชื่อ channel แบบ namespace:action เช่น db:getTasks, agentSession:start, worktree:setup, terminal:spawn

HTTP Servers ภายใน 3 ตัว (จุดที่น่าสนใจมาก)

Serverมีไว้ทำไม
task-api-server.tsให้ AI agent เรียกกลับเข้ามา (ผ่าน env TASK_API_URL) เช่น /update_task, /find_similar_tasks เพราะ agent เป็น subprocess แยก ใช้ IPC ไม่ได้ ต้องใช้ HTTP
mobile-api-server.tsให้มือถือ/เบราว์เซอร์เชื่อมเข้ามาดูงาน + WebSocket แจ้ง update real-time
secret-broker.tsแจก secrets ให้ agent ผ่าน per-session token (agent ไม่เห็น API key ตรงๆ)

13. Heartbeat Monitoring

หลัง agent ทำเสร็จ (เช่นเปิด PR แล้ว) งาน "ยังไม่จบจริง" เพราะต้องรอ CI, รอ review Heartbeat (heartbeat-scheduler.ts) คือระบบตามเช็คความคืบหน้าอัตโนมัติ:

  1. Agent เขียน heartbeat.md (checklist สิ่งที่ต้องตามเช็ค) ไว้ใน workspace
  2. เปิด heartbeat บน task → scheduler ตื่นทุก 60 วิ หา task ที่ถึงเวลาเช็ค
  3. spawn agent session เบาๆ ไปทำตาม checklist เช่น CI เขียวไหม? มี comment ใหม่ใน PR ไหม?
  4. บันทึกผลลง heartbeat_logs + แจ้งเตือนเมื่อต้องการความสนใจ
💡 Use Case จริง

Agent เปิด PR #123 ไว้ตอนเช้า คุณเปิด heartbeat แล้วไปทำอย่างอื่น บ่ายสอง CI ล้ม heartbeat ตรวจเจอแล้วส่ง notification เด้งบนเครื่อง: "PR #123 ของ task X มี CI failure" คุณกดเข้าไปสั่ง agent แก้ต่อได้ทันที ไม่ต้องนั่งเฝ้า GitHub

15. วิธีติดตั้งและรันโปรเจค

⚠️ ต้องมีก่อน

Node.js (LTS) + pnpm 9 (โปรเจคบังคับ pnpm ห้าม npm!) + Git และแนะนำ gh CLI (GitHub) / glab CLI (GitLab) สำหรับฟีเจอร์ worktree

# 1. ติดตั้ง dependencies (rebuild native modules อัตโนมัติผ่าน postinstall)
pnpm install

# 2. รันโหมดพัฒนา (hot reload)
pnpm dev

# 3. รัน tests / ตรวจ type / lint
pnpm test:run
pnpm typecheck
pnpm lint

# 4. Build แจกจ่าย
pnpm build:mac      # macOS (dmg + zip)
pnpm build:win      # Windows (NSIS)
pnpm build:linux    # Linux (AppImage)

# โหมด mobile web (แยกต่างหาก, port 5174)
pnpm dev:mobile

หมายเหตุ: better-sqlite3 และ node-pty เป็น native modules (C++) ต้อง compile ให้ตรงกับ Electron ถ้า build พังให้ดู scripts/rebuild-native.mjs

การใช้งานครั้งแรก

  1. เปิดแอป เจอ Onboarding Wizard ตรวจ CLI tools (git, gh, node, agent CLIs) ติดตั้งให้ถ้าขาด
  2. ไป Settings ตั้งค่า agent (เลือกค่าย AI + API key หรือ auth)
  3. เชื่อม integrations ผ่าน OAuth (Linear, Notion ฯลฯ)
  4. สร้าง/sync tasks แล้วกดปุ่ม Play (auto-run) หรือสั่งรันทีละงาน
